# Ведение проекта — как устроены переводы и как добавлять новые

Технический документ для дальнейшей работы над сборкой. Читать перед тем, как трогать переводы.

---

## 1. Как KSP находит локализацию

KSP грузит **все** файлы с блоком `Localization { <lang> { #key = value } }` из любой папки внутри `GameData` (обычно кладут в `GameData/<Mod>/Localization/ru.cfg`). Язык выбирается в игре или в `settings.cfg` строкой `LANGUAGE = ru` (тег именно `ru`, не `ru-ru`). Строки подтягиваются по ключам `#key` в реальном времени при отрисовке UI/деталей.

Из-за этого перевод — это **только добавление файлов**, оригиналы модов трогать не нужно.

## 2. Два типа модов

### Тип FULL — у мода есть своя система локализации

Внутри уже лежит `Localization/en-us.cfg` (или общий файл с языковыми блоками). Ключи вида `#autoLOC_...`, `#cc.*`, `#LOC_<mod>_*` и т.п. Достаточно положить рядом `ru.cfg` с блоком `Localization { ru { те же ключи = перевод } }`.

Наши FULL-моды (12): AJE, ContractConfigurator, EditorExtensionsRedux, ROCapsules, SXT, Trajectories, VenStockRevamp, KerbalChangelog, KerbalEngineerRedux, RealismOverhaul, SpaceTuxLibrary, BackgroundThrust.

### Тип PATCH — у мода локализации нет

Детали хранят текст прямо в полях `title = …`, `description = …` жёстко по-английски. Чтобы перевести, не трогая оригинал, кладём **два новых файла**:

1. `Localization/ru.cfg` — блок с `en-us` **и** `ru`:
   ```
   Localization {
     en-us { #LOC_<Mod>_<part>_title = <оригинал, побайтово> ... }
     ru    { #LOC_<Mod>_<part>_title = <перевод> ... }
   }
   ```
   Блок `en-us` **обязан точно повторять оригинальный текст** — это гарантирует, что для англоязычных игроков ничего не изменится.

2. `Localization/RuLocPatch.cfg` — ModuleManager-патч, переключающий поля деталей на ключи:
   ```
   @PART[<exact_part_name>] {
     %title       = #LOC_<Mod>_<part>_title
     %description = #LOC_<Mod>_<part>_description
     %manufacturer= #LOC_<Mod>_<part>_manufacturer
     %tags        = #LOC_<Mod>_<part>_tags
   }
   ```
   Оператор **`%`** = «установить или создать» — не удаляет и не ломает, если поля нет. Никогда не используй `@`/`!` здесь.

Наши PATCH-моды (8): ROSolar, ROTanks, ROEngines, ROHeatshields, ProceduralFairings, ProceduralParts, RealChute, ModularLaunchPads.

## 3. Правила перевода (обязательные)

1. **Терминология реальной космонавтики** (сборка = реальная Земля/Солнечная): apoapsis/periapsis → **апогей/перигей** (не «апоапсис»); thrust → тяга; stage → ступень; oxidizer → окислитель; nozzle → сопло; docking port → стыковочный узел; heat shield → теплозащитный экран.
2. **Длина ≤ оригинала.** Русский обычно длиннее — активно сокращай (аббревиатуры, выкидывание слов), иначе строка не влезет в UI. Правило: число слов в переводе ≤ числу слов в английском для **каждой** строки.
3. **Плейсхолдеры не трогать:** `<<1>>`, `{0}`, `%s`, `\n`, `[Что-то]` — сохранять как есть, в той же позиции.
4. **Комментарии `//`, скобки и пустые строки** оставлять как в оригинале.
5. **Имена-бренды не переводить:** `Rocketdyne`, `Thiokol`, `Generic`, названия программ.
6. **Служебные заглушки не переводить:** если в исходнике `title = abc` / `fixme` / `TBD` — оставить как есть в обоих языках (это заглушки автора мода, не текст).
7. **Функциональные теги не терять:** напр. `cck-modularlaunchpads` в `tags` читается CommunityCategoryKit для вкладки категории — сохранять дословно.

## 4. Как добавить перевод нового мода

1. Возьми исходник: `git clone` репозитория мода (надёжнее, чем GitHub API — git сам сверяет SHA).
2. Определи тип (есть ли `Localization/en-us.cfg`).
3. Извлеки строки/детали. **Парсер деталей должен понимать MM-операторы** в заголовке узла: `PART`, `+PART[base]:FIRST` (клон), `@PART[base]` — деталь считается реальной, если она задаёт `name` или `@name`. Учитывай, что поля бывают на разных уровнях вложенности — брать только прямые `title/description/manufacturer/tags` самой детали, не из вложенных `MODULE`.
4. Переведи по правилам §3.
5. Собери файл(ы) по §2 и положи в `GameData/<Mod>/Localization/`.
6. **Проверь** (§5) и протестируй запуском игры.

## 5. Проверка перед публикацией (не верь переводчику на слово)

Скрипт-парсер лежал в рабочей папке сессии (`verify_parts.py`); логика, которую он реализует:

- **Имена деталей** в патче существуют в исходнике (grep по `name = <part>` / `@name = <part>`).
- **en-us побайтово совпадает** с оригинальным текстом (иначе для англичан что-то изменится).
- **Длина ru ≤ en** по словам для каждой строки.
- **Баланс скобок** `{`/`}` в каждом файле 1:1.
- **Кодировка UTF-8**, кириллица не побилась при переносе (сверить размер в байтах на источнике и на цели).
- **Число ключей ru == en**, нет пропущенных/лишних.

⚠️ **Всё перечисленное выше — необходимо, но НЕ достаточно** (обнаружено 2026-07-22, см. [COVERAGE.md, раздел 0](COVERAGE.md#0-реальный-охват-по-деталям-партам--проверено-по-кэшу)). SXT/ROCapsules/VenStockRevamp/RealChute прошли ВСЕ эти проверки, но реально в игре показывают английский на 56-100% деталей. Единственная надёжная проверка — **сверка с `GameData/ModuleManager.ConfigCache` после реального запуска игры**: это уже полностью пропатченный конфиг, который игра использует напрямую.

```bash
# после хотя бы одного запуска игры с модом:
grep -A15 'name = <РеальноеИмяДетали>$' "GameData/ModuleManager.ConfigCache" | grep -E 'name =|title ='
# если title не содержит кириллицы — не работает, независимо от того, что в файлах.
```

Для проверки СРАЗУ всех деталей мода (не по одной) — см. `audit_ru_coverage.py` (упомянут в COVERAGE.md): парсит кэш в `name -> {title, description, manufacturer}`, сверяет со списком реальных имён деталей мода из исходников, считает % кириллицы по каждому полю.

## 6. Грабли, реально пойманные (важно!)

- **RP-1 не использует систему локализации для контрактов/техдерева/программ карьеры вообще** — `title`/`description`/`synopsis` зашиты прямо в `.cfg`, без единого `#LOC_`/`#autoLOC_` ключа. Добавить перевод в `ru.cfg` буквально некуда, пока не написан свой патч через `%title = #key`. У узлов с полем `name` (`CONTRACT_TYPE`, `RP0_PROGRAM`, `PARTUPGRADE`) — обычный `@NODETYPE[name] { %field = #key }`, подтверждено рабочим 2026-07-22.
- **`RDNode` не имеет поля `name`, только `id`** — синтаксис `@RDNode[value]` не сработает. Первая попытка `@RDNode:HAS[#id[value]]` (без скобки с именем) **выглядела корректной, MM её распознавал** (`Config(@RDNode:HAS[...])` в логе), но **ни разу не находил цель** — 0 строк `Applying update` против сотен для правильно сработавших узлов. Рабочий вариант: `@RDNode[*]:HAS[#id[value]] { %field = #key }` — обязательно нужна wildcard-скобка `[*]` перед `:HAS[]`. Проверяй именно по строке `Applying update ... to <файл>/<NODE>[<name>]` в логе, а не по факту, что MM вообще "увидел" патч — увидеть и применить это разные вещи.
- **Подсчёт `{`/`}` не ловит неправильную вложенность, только несовпадение количества.** При программной вставке новых ключей в СЕРЕДИНУ уже существующего файла (например, добавление ключей в чужой `ru.cfg`) — важно найти именно ПРАВИЛЬНУЮ закрывающую скобку, а не «первую при поиске с конца файла». Реальный баг 2026-07-22: скрипт нашёл внешнюю `Localization{}` вместо внутренней `ru{}`, и 252 ключа оказались вставлены вне блока `ru{}` (баланс скобок при этом остался 2/2 — проверка бессильна). Нашлось только построчным просмотром хвоста файла.
- **✅ РЕШЕНО 2026-07-22 (ночь): загадка SXT/RealChute — это гонка порядка проходов ModuleManager, не баг локализации.** RealismOverhaul сам патчит эти детали своими файлами совместимости (`RealismOverhaul/RO_SuggestedMods/SXT/RO_SXT_Utility.cfg`, `RealismOverhaul/RO_RecommendedMods/RO_RealChute.cfg`) — реалистично переименовывает их в `:FOR[RealismOverhaul]`, жёстко прописывая английский текст (`@title = PPD-4 Crew Cabin`, `%title = Parachute - Cone` и т.д.). Наши собственные патчи не указывали проход вообще (голый `@PART[x]{}` = проход Legacy, один из самых РАННИХ), а `:FOR[RealismOverhaul]` выполняется ПОЗЖЕ — значит патч RO применяется последним и затирает наш `#LOC_kerbalru_...`-редирект обратно на литеральный английский текст. Доказано побайтово: закэшированный `title` для `RC_cone` — это НЕ оригинальное название RealChute и не наш перевод, а именно текст ИЗ ФАЙЛА RealismOverhaul. То же самое объясняет недостающие проценты у ROTanks/ROSolar/ProceduralFairings/VenStockRevamp — RealismOverhaul патчит их выборочные детали точно так же (напр. `RSBRibbedBase_Boattail` в ProceduralFairings, подтверждено). ROCapsules — 100% потому что это RO-нативный мод, RealismOverhaul его не "довооружает" совместимостью, патчить нечего.
  **Фикс:** добавить `:FINAL` к каждому `@PART[...]` в наших `RuLocPatch.cfg` — это гарантирует, что наш патч применяется САМЫМ ПОСЛЕДНИМ, после вообще всех проходов (`:FIRST`/legacy/`:FOR`/`:BEFORE`/`:AFTER`/`:LAST`), включая любые будущие retrofit-патчи от других модов. Применено ко всем 12 файлам `RuLocPatch.cfg` (718 заголовков) 2026-07-22 — ждёт подтверждения следующим живым запуском.
  **Как искать подобное в будущем:** если патч подтверждённо применяется (`Applying update` в логе есть), структура верна, ключ определён с правильным значением — а в кэше всё равно английский текст, который НЕ совпадает ни с оригиналом мода, ни с нашим переводом, ни с сырым `#LOC_`-тегом — это почти наверняка третий патч откуда-то ещё, выполняющийся позже нашего. Ищи по имени детали во ВСЁМ `GameData` (`grep -rn '"@PART\[<имя>\]"'`), особенно в `RealismOverhaul/RO_SuggestedMods/` и `RO_RecommendedMods/` — RO часто патчит "неродные" (не свои) моды под реализм, включая `title`/`description`.
- **Contract Configurator запекает текст контракта в сейв** при генерации. Старые контракты в существующем сохранении останутся на английском — это не баг перевода. Проверять на новой карьере.
- **RP-1 патчит чужие Localization-блоки** через `RP-1/Localization/LocPatches.cfg` (целит стоковые `#autoLOC_*`, наши `#cc.*`/`#LOC_*` не задевает). Не пугаться строк `Applying update RP-1/... to <наш>/ru.cfg` в логе.
- **KerbalEngineerRedux**: правильная папка на диске — `GameData/KerbalEngineer/Localization/`, а не `GameData/Output/KerbalEngineer/…` (в репозитории структура иная).
- **SXT en-us.cfg**: несколько строк в исходнике склеены табами без переноса (`…_description = … \t\t #…_manufacturer = …`), а одна начинается с мусорного слова перед `#`. Наивный regex это ломает — парсить аккуратно.
- **ROEngines ключи**: при разбиении на части агент-переводчик может обрезать префикс `ROE-` в ключах (`Castor4` вместо `ROE-Castor4`). Проверять маппинг имён к исходнику.
- **Несбалансированные скобки в исходнике** отдельных `.cfg` (у 5 деталей в PF/MLP) — такие детали пропущены, оставлены на английском, чтобы не гадать. См. ROADMAP.
- **Большие моды** (SXT 804 строки, ROEngines 243 детали) не переводятся одним запросом — упираются в лимит вывода. Разбивать на части (у нас — 3–6 кусков), переводить параллельно, потом склеивать и проверять целостность ключей.
- **Регистр путей** на Linux/Deck важен (case-sensitive FS): имена папок модов сверять точно.
- **Ключ определён и в моде, и в нашем ru.cfg — а игра всё равно показывает третий английский текст.** Пример: SXT/`SXTSputnik`, ключ `#LOC.SXT_SXTSputnik_title` есть и в `en-us.cfg` мода, и в нашем `ru.cfg` (с настоящим переводом, не копией английского) — игра показывает `Sputnik PS-1`, что не совпадает НИ с одним из двух файлов. Проверено и исключено как причина: BOM/кодировка (везде честный UTF-8 без BOM), задвоение блоков `ru`/`en-us` в файле (по одному блоку на файл). Причина не найдена на 2026-07-22 — см. приоритет в ROADMAP.md. Если решите загадку — обязательно допишите сюда находку.

## 7. Где что лежит

- `GameData/` — готовые файлы локализации (структура повторяет GameData игры).
- `install-ru.sh` / `install-ru.ps1` — установщики только русификатора (поверх уже поставленной сборки).
- `install.sh` / `install.ps1` — установщик «всё в одном»: `--full`/`-Full` ставит сборку `RP-1-ExpressInstall` через headless CKAN и русификатор, `--ru-only`/`-RuOnly` делегирует в `install-ru.*`. Не хранит и не вендорит CKAN — только скачивает официальный релиз KSP-CKAN/CKAN.
- `data/project.json` — единый источник версии, состава сборки, общих и помодовых метрик покрытия; сайт читает его напрямую.
- `VERSION` — сгенерированная версия следующего стабильного релиза без префикса `v`; вручную не редактировать.
- `tools/sync_project_data.py` — проверяет JSON и обновляет README, `VERSION`, `CITATION.cff`.
- `tools/bump_release.py` — создаёт следующий номер формата `vYY.N` и запускает синхронизацию.
- `.github/workflows/release.yml` — проверяет совпадение тега с `VERSION` и создаёт GitHub Release.
- `index.html` + `vendor/three.min.js` — сайт (GitHub Pages). Three.js вендорнут локально (без CDN).
- `CNAME` — привязка домена kerbal.ru.
- `docs/` — эта документация.
- Иконки на сайте — Tabler Icons (MIT), встроены в `index.html`.

### Обновление сайта без правки HTML

Весь изменяемый контент сайта находится в `data/project.json`; `index.html` только строит интерфейс из этих данных:

| Что меняется | Поле JSON |
|---|---|
| версия, дата, канал и ссылка на релиз | `release` |
| версия KSP и профиль сборки | `game` |
| число компонентов и их категории | `inventory` |
| общие проценты и активный тип аудита | `coverage` |
| помодовые числа деталей | `mods` |
| заголовки, тексты основных секций и подписи CTA | `site.content` |
| два сценария установки, схема процесса и пошаговые инструкции для ОС | `site.installation` |
| карточки поддерживаемых модов, описания, ссылки и скриншоты | `site.modCatalog` |
| остальные компоненты единого каталога | `site.inventoryGroups` |
| основные моды, показанные в каталоге по умолчанию | `featured: true` у карточки или записи инвентаря |
| GitHub-участники и их роли | `site.contributors` |
| участники и пустое состояние скриншотов по умолчанию | `site.modDefaults` |
| устройства, методика и результаты замеров производительности | `site.performance` |
| перевод окон/кнопок через Harmony | `uiTranslation` |
| компоненты, впервые получившие русский через UI-перевод | `uiTranslation.inventoryAdditions` |
| ссылки сайта | `links` |

Чтобы добавить поддерживаемый мод, добавь его метрики в `mods` и одну карточку с тем же `id` в `site.modCatalog`, затем обнови `inventory.kerbalRuMods`. Чтобы изменить полный состав, редактируй массив `entries` нужной группы в `site.inventoryGroups` и соответствующий счётчик в `inventory`. Все группы отображаются в одном поисковом каталоге; отдельного визуального списка «кандидатов» больше нет.

Каждый сценарий в `site.installation.paths` содержит общую схему `flow` и
четыре инструкции в `platforms`: `windows`, `macos`, `linux`, `deck`. Внутри
платформы основные действия задаются массивом `steps`; запасной ручной путь —
объектом `manual` с собственными `steps`. Шаг может содержать `title`, `text`,
`code` и `links`. Для ссылки на ZIP текущего релиза используй
`"releaseDownload": true` вместо зашитого номера версии. Команды установки и
флаги меняются только здесь: `index.html` строит переключатели платформ,
нумерацию, блоки команд и ручной способ автоматически.

У каждой карточки можно задать:

```json
{
  "featured": true,
  "contributors": ["plagness"],
  "screenshots": [
    {
      "src": "assets/screenshots/mod-id/editor.webp",
      "alt": "Мод в редакторе ракеты",
      "caption": "Короткая подпись"
    }
  ]
}
```

`contributors` ссылается на `site.contributors[].id`; аватар и профиль строятся из
GitHub-логина. Если `screenshots` пуст, сайт показывает аккуратное место под
будущий материал. Те же необязательные поля можно добавлять записям
`site.inventoryGroups[].entries`. `featured: true` включает компонент в
короткий стартовый список «Основные»; полный состав всегда доступен по кнопке
«Все». `uiTranslation.inventoryAdditions` содержит только точные названия
компонентов из инвентаря, которые впервые получили русский через Harmony-патч:
сайт прибавляет их к общему охвату без двойного счёта.

Для замера производительности сначала используй шаблон GitHub Issue
`performance.yml`, проверь воспроизводимость и согласие автора, затем перенеси
результат в `site.performance.devices`: поставь `status: "measured"` и заполни
версию сборки, разрешение, пресет, средний/1% low FPS, пиковую RAM и время
загрузки. Сайт не должен собирать телеметрию или отпечаток устройства в фоне.

После любой правки выполни:

```bash
python3 tools/sync_project_data.py --write
python3 tools/sync_project_data.py --check
```

Первая команда синхронизирует README, `VERSION` и `CITATION.cff`; вторая проверит проценты, полноту каталога, число элементов в группах и обязательные поля. Если JSON корректен, GitHub Pages подхватит значения без правки `index.html`.

## 8. Релиз и обновление

Установщики по умолчанию читают последний GitHub Release. В каталоге KSP они создают:

- `.kerbalru-version` — установленная версия;
- `.kerbalru-files` — manifest файлов, которыми управляет проект;
- `kerbal.ru-backups/` — резервные копии перед заменой или удалением managed-файла.

Обычный релизный цикл:

1. если есть свежий игровой кэш, выполнить `python3 tools/audit_ru_coverage.py --write-project-data`, затем `python3 tools/sync_project_data.py --write`;
2. выполнить `python3 tools/bump_release.py` (например, `v26.1` → `v26.2`) и добавить запись в `CHANGELOG.md` (секция `[Unreleased]` → новая версия);
3. проверить чистую установку и обновление поверх предыдущего релиза на обеих реализациях установщика;
4. влить релизный PR в `main`;
5. создать и отправить тег `v$(cat VERSION)` — workflow выпустит GitHub Release;
6. **обязательно написать содержательное описание релиза** (`gh release edit vX.Y --notes "..."`, разделы «В релизе» / «Текущие показатели» / «Дальше», по образцу v26.1/v26.2) — workflow сам вызывает `--generate-notes`, это ТОЛЬКО список коммитов, не описание для пользователя; без этого шага релиз выглядит пустым;
7. проверить `--check` / `-Check` и обычное обновление через публичный `kerbal.ru`.

Никогда не переиспользуй и не передвигай опубликованный тег. Любое исправление уже выпущенного релиза получает следующий номер текущего года. Полная пользовательская инструкция — в [UPDATING.md](UPDATING.md).

## 9. Тестовая сборка (референс)

Рабочая сборка стоит на Steam Deck: `~/.local/share/Steam/steamapps/common/Kerbal Space Program`.
CKAN на Deck запускается из контейнера **distrobox `ckanbox`** (Debian + `mono-complete`), т.к. SteamOS read-only. Лог игры: `KSP.log` в корне установки — грепать по `Config(Localization)` (загрузка), `[ERR`/`Exception` (проблемы).
