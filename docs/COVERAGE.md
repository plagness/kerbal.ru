# Покрытие переводом — реальные цифры (проверено по игровому кэшу)

⚠️ **Обновлено 2026-07-23.** Раздел 0 ниже — цифры, проверенные по `ModuleManager.ConfigCache` (то, что реально загружает игра), а не по факту наличия файла локализации. Более ранняя методика («файл `ru.cfg` есть → мод переведён», разделы 1-3 ниже) **переоценивала готовность**: файл может лежать в папке и даже содержать правильный ключ, а игра всё равно показывает английский. Разделы 1-3 сохранены как raw-данные по наличию файлов, но не считайте их показателем реальной русификации — доверяйте разделу 0.

## 0. Реальный охват по деталям (партам) — проверено по кэшу

Метод: для каждого из 20 модов взят список реальных `name` деталей из исходников мода, каждое имя сверено с полями `title` / `description` / `manufacturer` в финальном `GameData/ModuleManager.ConfigCache` — это уже полностью пропатченный конфиг, который игра использует напрямую. Наличие кириллицы = деталь реально на русском в игре; отсутствие = не работает независимо от того, что лежит в файлах.

**RealChute — полностью решено.** Все 4 парта, которые патчились через наш обходной `%title = #LOC_kerbalru_...`, теперь корректно показывают русский — фикс `:FINAL` (гарантирующий, что наш патч выполняется после ретрофит-патчей RealismOverhaul на проходе `:FOR[RealismOverhaul]`) сработал именно так, как предполагалось.

**VenStockRevamp — найдена и устранена настоящая причина «26% вместо 41%».** У мода уже есть **полная нативная русская локализация** (90/90 title, 85/85 description — 100% того, что переводимо), она не сломана. Проблема была в других 31 деталях: `RealismOverhaul/RO_SuggestedMods/VSR/VNP/*.cfg` жёстко переписывает их `title`/`description` на английский на проходе `:FOR[RealismOverhaul]`. Наш обходной `:FINAL`-патч прикрывал только 15 из этих 31 — остальные 16 (`OscarA/D/Etank`, `RadialLF(O)(Long)`, `CryoXBig/med/small` и др.) добавлены сегодня, ждут подтверждения следующим запуском.

**SXT — частично решено, обнаружена вторая, более крупная причина.** SXT — «полный»-тип мод: у него 208 партов, но нашим обходным патчем покрыто только 60 (`kerbalru-fix.cfg`) — эти 60 теперь резолвятся правильно (подтверждено побайтово: например `SXTSputnik` → title `"Sputnik PS-1"` (намеренно оставлено, историческое обозначение), description `"Первый спутник Земли на орбите."`, manufacturer `"ОКБ-1 (Королёв)"` — всё верно). Остальные **148 партов** используют **родную** систему локализации SXT (свой `Localization/ru.cfg`, 810 строк, реально переведён и НЕ сломан сам по себе — проверено на `GearFixed`: title `"Фиксированное шасси LY-01"`, manufacturer `"Машилин Тайрз"`, оба на русском). Но 143 из этих 148 **вообще не появляются в финальном кэше под своим именем**. Расследование показало главную причину:

> **RP-1 сам вырезает детали без флага `RP0conf = True` через собственный фильтр `HardRemoveNonRP0`** (`RP-1/NonRP0/…`, проход `:BEFORE[zzzTagCleanup]:NEEDS[HardRemoveNonRP0]`) — это «режим жёсткого реализма» RP-1, удаляющий любые «не-канонiчные» детали, включая обычные SXT-детали без RP-1-специфичной конфигурации. Проверено на `SXT16FuelTank`: в логе — `Applying delete RP-1/NonRP0/!PART[*]:HAS[~RP0conf[*rue]]…`. Это **не баг и не наш патч** — осознанная функция самого RP-1, включённая в этой сборке.

**Практический вывод:** заявленные «208 партов SXT» и «~215 title ROEngines/аналоги» у других модов может систематически завышать реальный объём — часть этих партов **физически не существует** в игре при текущих настройках RP-1, независимо от перевода. Процент 27% для SXT — это НЕ «73% реально показывают английский», а смесь «часть уже переведена и работает» + «часть удалена RP-1 и не показывается вообще» + возможно небольшой реальный остаток, который ещё предстоит вычленить отдельным аудитом (не делали в этом проходе — не выявили ни одного НАСТОЯЩЕГО английского title среди проверенных живых партов SXT, что обнадёживает).

### Находка: неправильный синтаксис MM-селектора для RDNode (решалось в 2 захода)

**Первая попытка (2026-07-22).** `RDNode` — единственный из этих узлов без поля `name` (только `id`), поэтому потребовался особый селектор `:HAS[]`. Версия `@RDNode:HAS[#id[x]]` **не сработала вообще** (0 из 332) — лог показал, что MM **распознал** все 332 патча, но ни разу не нашёл цель. Заменено на `@RDNode[*]:HAS[#id[x]]` (с wildcard-скобкой) — казалось логичным (раз нет `name`, нужен wildcard), но при живой проверке 2026-07-23 **снова 0 из 332 "Applying update" в логе**, несмотря на то, что фикс был признан рабочим и попал в релиз v26.2/26.3.

**Настоящая причина (найдена 2026-07-23).** `RDNode` — это **не корневой тип узла конфига вообще**, а вложенный дочерний узел внутри `@TechTree:FOR[ZZZZZ-RP0Tree] { RDNode {...} RDNode {...} ... }`. Любой селектор вида `@RDNode[...]` на верхнем уровне ищет узел `RDNode` **среди корневых узлов всей базы данных игры** — таких не существует, они все вложены в `TechTree`. Отсюда «распознано, но не применено» — синтаксически патч валиден, но цели для него в принципе нет на верхнем уровне.

**Реальный фикс**: патч должен быть тоже вложенным — редактировать сам `TechTree`, а внутри него, уже в контексте дочерних узлов, ссылаться на `RDNode` обычным (не wildcard) селектором:
```
@TechTree:FINAL
{
	@RDNode:HAS[#id[start]]
	{
		%title = #LOC_kerbalru_rp1_tech_start_title
		%description = #LOC_kerbalru_rp1_tech_start_description
	}
	... (332 таких блока)
}
```
`:FINAL` на самом `@TechTree` гарантирует, что наш патч видит дерево уже полностью собранным RP-1 (который сам патчит `TechTree` на именованном проходе `:FOR[ZZZZZ-RP0Tree]`, идущем раньше `:FINAL`). Применено 2026-07-23, ждёт подтверждения следующим живым запуском.
