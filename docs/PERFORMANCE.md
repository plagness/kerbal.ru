# Производительность сборки RO/RSS/RP-1 (KSP 1.12.5)

> ЧЕРНОВИК. Синтез пяти разведотчётов по реальной установке на Mac M1 Pro 16 ГБ.
> Проверено по `KSP.log` и `ModuleManager.log`. Цифры платформозависимы — используй как
> ориентир, не как гарантию. Steam Deck — по архитектуре и отчётам сообщества (живой замер
> ещё предстоит).

Этот документ описывает **почему сборка хромает** и **что с этим делать** — от бесплатных
правок настроек до платформенных трюков. В конце — дизайн интеграции «профиля
производительности» в автоустановку kerbal.ru.

Философия проекта здесь та же, что в `FOR-AGENTS.md`: **мы не распространяем чужие моды** —
перф-моды (MemGraph и т.п.) ставятся тем же headless-CKAN, который уже вызывает `install.sh`;
а **свои конфиги** (MM-патчи графики/физики, тюнинг padheap) распространять можно — они наши.

## 0. TL;DR — что делать прямо сейчас

Ранжировано по «выигрыш ÷ усилие», сверху — самое выгодное и безопасное:

1. **Второй запуск уже быстрый.** 6 минут — это разовая сборка кэша ModuleManager после
   установки/обновления русификатора. Не переустанавливай перевод перед каждой игрой — и
   загрузка падает 360 с → ~150 с сама. Подробности → §4.
2. **RSS-Textures 8K → 4K** (Mac) / **→ 2K** (Deck) через CKAN. −1 ГБ RAM/VRAM, меньше фризов
   при смене тела. На экране почти без разницы (см. §2.3). Риск ноль.
3. **`SHADOWS_QUALITY` 4 → 2 (Mac) / 1 (Deck)** и **Scatterer тени 8192 → 2048/1024**. Крупный
   GPU-выигрыш там, где сейчас «хромает» — вид на поверхность Земли и КЦ.
4. **Поставить HeapPadder** (или MemGraph) через CKAN + наш урезанный `padheap.cfg` (~512–768 МБ)
   — против периодических GC-фризов от Kerbalism. Кросс-платформенно.
5. **Deck:** кап FPS 30, TDP 15 Вт, Scatterer-океан off, вырезать объёмные облака EVE.

Всё это можно применить одной командой — см. §6 (`install.sh --perf`).

## 1. Диагноз: почему хромает

KSP1 делит нагрузку на **две несмешиваемые части**, и лечатся они по-разному:

| Часть | Что в неё входит | От чего зависит |
|---|---|---|
| **CPU / один поток** | загрузка (патчинг ModuleManager), физика аппаратов, фон Kerbalism | почти НЕ зависит от платформы/графики; лечится кэшем MM, числом партов, текстурами (память) |
| **GPU** | просадки при виде на поверхность Земли/КЦ (Scatterer, EVE, RSSVE) | решает платформа и графический API; здесь узкое место Mac |

### Mac M1 Pro (16 ГБ) — боль = Rosetta + время загрузки
- Бинарник KSP — **Mach-O x86_64**, нативной ARM-сборки KSP1 нет → игра идёт под **эмуляцией
  Rosetta 2**. Rosetta транслирует x86→ARM один раз (AOT, кэшируется), CPU-код после этого
  ~70–80% нативного — то есть Rosetta **не** главный тормоз.
- Главный тормоз загрузки — **однопоточный патчинг ModuleManager** (CPU-bound, под Rosetta
  медленный) и то, что он **пересобирается каждый раз, когда меняются `.cfg`** (в т.ч. наши).
- GPU-статтер в атмосфере — потому что macOS-сборка рендерит через **устаревший Apple OpenGL
  4.1** (Metal у KSP нет), а GL-драйвер на macOS однопоточный и медленный.
- 16 ГБ **общей** памяти делятся с системой → 8K-текстуры + GameData 7.6 ГБ создают memory
  pressure и своп.

### Steam Deck (SteamOS, RDNA2 APU, 16 ГБ shared) — боль = GPU + общая память
- Здесь **нет Rosetta** (нативный Linux x86-64), MM-патчинг и Burst идут быстрее.
- Узкое место — **GPU** (15 Вт APU, fill-rate/шейдеры) и **отсутствие выделенной VRAM** (всё в
  тех же 16 ГБ).
- Подтверждённый **баг DXVK #5140**: на больших модпаках RSS/RO текстуры переполняют GTT-память
  → **150 FPS падают до 15** на некоторых дистанциях. Через Proton (DXVK) 8K RSS почти
  гарантированно ловят overspill. Лечится **только уменьшением текстур** (флаги
  `RADV_PERFTEST`, AMDVLK в багрепорте не помогли).

**Вывод:** на Mac бьём по загрузке (кэш MM) и памяти (текстуры), потом по GPU (тени/Scatterer);
на Deck — агрессивно режем GPU и память, каппим кадры и мощность.

## 2. Настройки — таблицы значений для двух профилей

Все правки ниже — **наши собственные конфиги / стоковые настройки** (не чужие моды), их можно
включить в автоустановку двумя профилями `mac` / `deck`.

Важный факт: `TEXTURE_QUALITY = 2` (Quarter) — это глобальный mip-bias (Unity
`masterTextureLimit`), он **уже даунсемплит 8K-текстуры RSS до ~2K в VRAM**. То есть за 8K-пак
ты платишь RAM и временем загрузки, а видишь ~2K. Это главный аргумент за переход на 4K-пак:
на экране почти без разницы, а память/загрузка — вдвое легче.

### 2.1. Стоковые графические настройки — `settings.cfg`

Файл: `<KSP>/settings.cfg`. **Патчить только при закрытой игре, до запуска** (игра
перезаписывает файл на выходе — механику см. §6.3).

| Параметр | Сейчас | Mac | Deck | Выигрыш | Что теряется |
|---|---|---|---|---|---|
| **SHADOWS_QUALITY** | **4** | **2** | **1** (или 0) | **Высокий (GPU)** | грубее/мягче тени; при 0 теней нет (плоско, макс. FPS) |
| TEXTURE_QUALITY | 2 | 1* | 2 | Средний (VRAM/RAM/load) | *с 4K-паком (см. 2.3) Mac→1 даёт чёткие части при том же бюджете; Deck держать 2 |
| ANTI_ALIASING | 0 | 0 | 0 | — | уже выкл; лесенки гасит SMAA Scatterer |
| LIGHT_QUALITY | 8 | 8 | 4 | Низкий | меньше одновременных пиксельных источников света |
| AERO_FX_QUALITY | 3 | 2 | 2 | Низкий | менее детальные эффекты входа/плазмы |
| TERRAIN_SHADER_QUALITY | 0 | 0 | 0 | — | уже минимум (Low); Ultra-шейдер террейна очень дорогой — держать Low |
| PLANET_SCATTER | False | False | False | — | уже выкл (камни/трава). Держать off |
| REFLECTION_PROBE_TEXTURE_RESOLUTION | 1 | 1 | 0 | Низкий | менее детальные отражения на корпусе |
| CELESTIAL_BODIES_CAST_SHADOWS | True | True | False | Низкий | Deck: пропадут тени затмений (косметика) |
| SHADOWS_MAIN_PROJECTION | 1 | 1 | 0 | Низкий | тени в редакторе/меню (не по геймплею) |
| SHADOWS_EDITORS_PROJECTION | 1 | 0 | 0 | Низкий | то же |
| SYNC_VBL | 1 | 1 | 1 | — | Vsync; на Deck лучше кадровый лимитер Steam, а не игры |
| FRAMERATE_LIMIT | 60 | 60 (или 30) | **30–40** | Средний | Deck: ровнее frametime, меньше тепла/жора |
| QUALITY_PRESET | 1 (Low) | 1 | 1 | — | обёртка над вышеперечисленным — держать |

### 2.2. Физика — `PHYSICS_FRAME_DT_LIMIT` (в `settings.cfg`)

Это «Max Physics Delta-Time per Frame» (Unity `Time.maximumDeltaTime`): **не** меняет шаг физики,
а ограничивает, сколько «догоняющих» физ-шагов игра втискивает в медленный кадр.

| Параметр | Сейчас | Mac | Deck | Эффект | Что теряется |
|---|---|---|---|---|---|
| **PHYSICS_FRAME_DT_LIMIT** | **0.04** | **0.03** | 0.03 | стабильнее физика, меньше вобла (особенно Mac/Rosetta) | под тяжёлой нагрузкой сильнее slow-mo, но ракеты выживают |

Безопасный диапазон для RO — **0.03–0.05**. Ниже 0.03 не надо (чрезмерный slow-mo). Стоят KJR +
KSPCF (`AutoStrutDrift`, `PartStartStability`, `LowerMinPhysicsDTPerFrame`) — 0.03 безопасно.

### 2.3. RSS-Textures 8K → 4K/2K (через CKAN, не файлами)

`TEXTURE_QUALITY=2` и так показывает ~2K — так что переход почти незаметен на глаз, но
загрузка/RAM/VRAM легче. Варианты — **отдельные CKAN-пакеты**, все предоставляют один
виртуальный `RSSTextures`, от которого зависит RealSolarSystem, поэтому замена чистая.

| Пак | CKAN id | На диске | Разрешение Земли | Кому |
|---|---|---|---|---|
| 8K (сейчас) | `RSSTextures8192` | ~1.6 ГБ | 8192×4096 | — |
| **4K** | **`RSSTextures4096`** | ~0.8 ГБ | 4096×2048 | Mac 16 ГБ (рекоменд.) |
| 2K | `RSSTextures2048` | ~0.3 ГБ | 2048×1024 | Deck / при OOM |

```bash
ckan remove  --headless RSSTextures8192      # порядок remove→install важен
ckan install --headless RSSTextures4096      # CKAN сам разрулит "оба provide RSSTextures"
```

Эффект: ассеты (сейчас 52 с) грузятся ~вдвое быстрее; RAM/VRAM под планеты ~вдвое меньше;
**на Deck дополнительно обходит баг GTT-overspill #5140**. На Mac после 4K можно вернуть
`TEXTURE_QUALITY=1` — 4K даунсемплится до ~2K в VRAM (тот же бюджет памяти, но части чётче).

⚠️ **Внимание:** сами текстуры мы НЕ распространяем — это делает официальный CKAN. Наш скрипт
только вызывает `ckan remove/install`.

### 2.4. Scatterer — GPU-тяжёлые эффекты (наш MM-патч)

Текущий конфиг близок к «Very High» (FFT-сетка 128, тени 8192). Правим через **свой MM-патч**
(не редактируя `Scatterer/config/config.cfg` напрямую):

| Параметр | Сейчас | Mac | Deck | Выигрыш | Что теряется |
|---|---|---|---|---|---|
| **unifiedCamShadowResolutionOverride** | **8192** | **2048** | 1024 / 0 | **Высокий (GPU+VRAM)** | менее чёткие тени облаков/рельефа — один из самых дорогих параметров всей сборки |
| useOceanShaders | True | True | **False** | **Высокий на Deck** | Deck: плоский цвет воды без волн/пены/преломления |
| m_fourierGridSize | 128 | 64 | 64 | Средний | грубее геометрия волн |
| oceanCaustics / oceanLightRays | True | False | False | Низкий-средний | нет каустик/подводных лучей |
| oceanFoam / oceanTransparencyAndRefractions | True | True/True | False/False | Средний (Deck) | Deck: нет пены/прозрачности воды |
| useLowResolutionAtmosphere | False | False | **True** | Средний (Deck) | half-res рассеивание, мягче градиенты неба |
| useTemporalAntiAliasing | True | True | False | Низкий | Deck: чуть больше мерцания краёв, но дешевле |
| smaaQuality | 2 | 1 | 1 | Низкий | чуть больше лесенки |
| useEclipses / useRingShadows | True | True | False | Низкий | Deck: без затмений/тени колец |
| **terrainShadows** | **False** | False | False | — | уже off — очень дорого, **держать выключенным** |

**Альтернатива без патча (ручная):** в игровом меню Scatterer выбрать пресет — Mac → «Low»,
Deck → «Integrated Graphics» (океан-шейдеры off целиком). Пресеты в
`Scatterer/config/qualityPresets.cfg`. Но `unifiedCamShadowResolutionOverride` пресеты Low/Optimal
**не переопределяют** — его надо задавить вручную/патчем в любом случае.

### 2.5. EVE / RSSVE — объёмные облака (наш MM-патч)

Это классический **EVE Redux** (не платные volumetric-облака Blackrack). «Объёмность» — блоки
`layerVolume` (3D-облака, сквозь которые летишь) у Земли, Марса, Титана, Венеры. Именно они дают
просадку на малой высоте (овердро частиц) — как раз на взлёте/посадке.

| Элемент | Сейчас | Mac | Deck | Выигрыш | Что теряется |
|---|---|---|---|---|---|
| **layerVolume (Earth/Mars/Titan/Venus)** | вкл | оставить | **вырезать** | Средний-высокий на малой высоте | пропадают 3D-«клубы» вблизи, остаётся 2D-слой |
| shadowMaterial `_ShadowFactor` | 0.6–0.8 | оставить | 0 | Низкий | нет тени облаков на земле |
| Вариант RSSVE | RSSVE-LR | ок | ок | — | уже низкотекстурный вариант — хорошо |

MM-патч, вырезающий `layerVolume` у этих тел, — **наш файл**, распространять можно.

### 2.6. Kerbalism — фоновая симуляция

Уже настроен грамотно (`Reliability=false`, `SpaceWeather=false`, `StormFrequency=0`,
`UseSamplingSunFactor=true`). Главная нагрузка — фоновый пересчёт ресурсов по **всем** аппаратам
каждый тик; рычаг не в конфиге, а в дисциплине: чистить обломки, держать флот компактным.

| Параметр | Сейчас | Mac | Deck | Выигрыш |
|---|---|---|---|---|
| LowQualityRendering | false | false | true | Низкий (Deck: меньше частиц магнитосферы) |
| MAX_VESSELS_BUDGET | 250 | 250 | 100–150 | если накопилось много debris |

## 3. Моды к добавлению и патчи KSPCommunityFixes

### 3.1. Перф-моды через CKAN

Ставятся **тем же headless-CKAN**, который уже вызывает `install.sh`. Мы их не храним.

| Мод | Что делает | Выигрыш | CKAN id | Mac (Rosetta) | Deck | Риск |
|---|---|---|---|---|---|---|
| **HeapPadder** | пэддинг Mono-heap → GC реже, без UI, **автопад на загрузке** | микрофризы (GC-stutter) | `HeapPadder` | ✅ чистый C# | ✅ | почти ноль; ест ~0.5–0.7 ГБ RAM |
| **MemGraph** | то же + график alloc rate; **пэддинг по хоткею `Mod+End`** | микрофризы + диагностика | `MemGraph` | ✅ | ✅ | почти ноль; для автоматики хуже (нужен хоткей) |
| KSPTextureLoader | async-загрузка текстур off-main-thread (BC7, AVX2 через Burst) | время загрузки, микрофризы | `KSPTextureLoader` | ⚠️ AVX2 под Rosetta — тестировать | ✅ | молодой мод; ставить опц. и изолированно |
| KSPBurst / -Lite | Burst-компиляция Unity-job'ов | FPS **только** если мод юзает Burst-jobs | `KSPBurst` / `KSPBurst-Lite` | ⚠️ профита ~0, +время 1-й загрузки | ✅ (быстрее чем Mac) | оставить как зависимость, в перф не выносить |

**Ключевой выбор — HeapPadder vs MemGraph:** механизм один (аллоцируют большие managed-массивы,
чтобы Mono вырастил heap → GC срабатывает реже → меньше stop-the-world фризов). Оба — **чистый
C#, нативной библиотеки нет** → работают одинаково на Win/Linux/macOS-Rosetta. Ставить **одно из
двух**, не оба.
- Для **автоустановки** — `HeapPadder`: пэддит автоматически на загрузке сцены, хоткей не нужен.
- Для ручного тюнинга — `MemGraph`: есть график (`Mod+KeypadMultiply`), пэддинг `Mod+End`.

**Наш конфиг padheap** (распространять можно — это наш файл): на 16 ГБ рядом с RSS дефолтный
~1024 МБ пэддинг жирный (толкает в своп). Ставим **512–768 МБ**. Кладётся в
`GameData/<HeapPadder|MemGraph>/PluginData/.../padheap.cfg` формата «размер блока : число блоков»,
**после** CKAN-установки мода. Точное значение подбирать по alloc rate (Kerbalism + много
аппаратов = высокий rate → мод максимально в тему).

⚠️ **Не «больше — лучше»:** избыточный пэддинг на 16 ГБ толкает в своп → станет хуже. Начинать с
~512 МБ.

### 3.2. KSPCommunityFixes — что доключить (наш MM-патч)

Все перф-патчи **уже включены** (SceneLoadSpeedBoost, FastLoader, PQSUpdateNoMemoryAlloc,
MemoryLeaks, DisableManeuverTool=true, LocalizerPerf, FlightIntegratorPerf, CollisionManagerFastUpdate
и ~35 других) — трогать не надо. Выключены по умолчанию, стоит включить:

| Патч | Сейчас | Рекоменд. | Зачем |
|---|---|---|---|
| **FewerSaves** | false | **true** | не пишет автосейв при выходе из мелких зданий КЦ — заметно меньше stutter в KSC-сцене |
| **SkipIndentsOnSavesAndCraftFiles** | false | **true** | быстрее запись сейвов/крафтов — меньше стоп-кадров при автосейве на больших сейвах RP-1 |
| AsteroidAndCometDrillCache | false | — | нишевое (астероидный майнинг), на RP-1 обычно не нужно |

Менять **не** редактированием `KSPCommunityFixes/Settings.cfg` (слетит при обновлении мода), а
нашим MM-патчем (`:FINAL`, чтобы перебить дефолт):

```
@KSPCommunityFixes:FINAL
{
    @FewerSaves = true
    @SkipIndentsOnSavesAndCraftFiles = true
}
```

`DisableManeuverTool=true` уже стоит — правильно, один из главных источников фризов при работе с
узлами манёвров на Kopernicus-системах (RSS). Не выключать.

### 3.3. Аудит сборки — что снять (опционально, через CKAN)

Хорошая новость: **Principia, SCANsat, Parallax, TUFX — НЕ стоят** (главные пожиратели RAM/CPU
отсутствуют, чёрный список RP-1 чист).

Кандидаты на удаление (не ядро RP-1, снижают и RAM, и число MM-патчей → и время загрузки):

| Мод | Вес | Вердикт |
|---|---|---|
| **SXT** | 99 МБ | доп. парт-пак, RP-1 не нужен — снять (`ckan remove --headless SXT`) |
| **VenStockRevamp** | 91 МБ | перекрыт ReStock, RO прячет стоковые парты — снять |
| ReStock + ReStockPlus | ~755 МБ | оверхол стоковых партов; в RP-1 летаешь на RO-*, стоковые скрыты. **Снять после теста** на розовые/пропавшие модели (RO рекомендует ReStock — проверять) |
| ModularLaunchPads | 328 МБ | косметические башни/пады, RP-1 работает без него — кандидат при RAM-голоде |

Крупные, но **НЕ в RAM** (не трогать): `000_KSPBurst` (391 МБ — тулчейн), `KSPCommunityFixes`
(140 МБ — обязателен). Ядро RP-1 (ROTanks/ROEngines/ROCapsules ~2.5 ГБ, FAR, RealFuels,
Kopernicus, RealSolarSystem, RealismOverhaul) — **оставить**.

**Дисциплина part-count** (совет игроку): строй баки/обтекатели/крылья **процедурными**
(ProceduralParts/Fairings/Wings, ROTanks) вместо стыковки многих мелких → меньше джойнтов → выше
FPS, меньше нагрузка Kerbalism, меньше MM-записей.

## 4. Время загрузки: 360 с → ~150 с

Источник: `KSP.log` → `Total loading time to main menu : 359.784s` (`Apple M1 Pro | 16384 MB`).

### 4.1. Раскладка 360 с по стадиям (из лога)

| # | Стадия | Время | Что происходит |
|---|---|---|---|
| 1 | движок + DLL модов | ~8 с | Unity/Mono, ~100 ассемблий |
| 2 | GameDatabase — чтение ассетов | **~52 с** | `Assets loaded in 52.497s`; текстуры RSSVE/Scatterer/партов. Работает KSPCF **FastLoader** (уже вкл) |
| 3 | **ModuleManager патчинг** | **~185 с** | `Waited 184.943s for patching`; **78181 патч вживую**, `useCache = False`. ГЛАВНОЕ |
| 4 | применение БД + post-patch | ~11 с | `Post patch ran in 10.374s` |
| 5 | PartLoader (парты + drag cubes) | **~47 с** | 1241 парт + 1029 drag cubes пересчитаны заново |
| 6 | планетарка Kopernicus + сцена меню | **~62 с** | из них `Planetary system loaded in 33.696s` |

> Замечание о цифрах: ранняя грубая оценка «~344 с патчинга» объединяла стадии 3+5 (+часть 4/6),
> которые каскадом текут из одного промаха кэша. Лог-точная разбивка — выше: сам MM-патчинг ~185 с.

### 4.2. Корневая причина «каждый раз долго» — инвалидация ConfigCache нашими `.cfg`

`ModuleManager.log` пишет прямым текстом:

```
Added : ContractConfigurator/Localization/zzz-kerbalru-cc-fix.cfg
Added : RP-1/Localization/kerbalru-ccgroups-fix.cfg
Added : RP-1/Localization/kerbalru-ccgroups-patch.cfg
useCache = False
```

Три **наших** файла (`kerbalru-*`), добавленные с прошлого запуска, изменили SHA всей
конфиг-базы → `useCache=False` → полный перепатч 185 с. Каскадом невалиден `PartDatabase.cfg` →
1241 парт и 1029 drag cubes пересчитаны заново (+~30 с).

**НО** в конце этого запуска кэш пересобран и сохранён (`Saving Cache`). Значит:
- **не переустанавливать русификатор перед стартом → следующий запуск быстрый** (кэш-хит, стадия
  3 схлопывается в ~5–10 с);
- **если workflow перед каждой игрой заново прогоняет `install-ru.sh` с изменением `.cfg` → кэш
  бьётся каждый раз → 185 с каждый раз.** Это и есть «каждый раз долго» в дев-цикле.

**Что инвалидирует кэш, а что нет:**
- MM хэширует **содержимое `.cfg`-файлов + версии сборок** (по контенту, не по mtime).
- **`.txt`-словари MM НЕ хэширует** → синк `.txt` **безопасен**.
- Побайтово одинаковый `.cfg` **не** бьёт кэш. Инвалидация только при реальном изменении контента
  `.cfg` или добавлении/удалении `.cfg`.
- Наши loc-патчи — `.cfg` с нодами `Localization{}`, поэтому именно они и есть источник.

Доп. фактор: наши патчи тяжёлые сами по себе — **~1013 патчей, из них 692 `:FINAL`** (~27% самого
дорогого финального прохода). Раздувают и холодный 185 с.

### 4.3. Действия и экономия

| Действие | Экономия | Примечание |
|---|---|---|
| **Не инвалидировать MM-кэш перед запуском** (синкать только `.txt`) | **−185 с** каждый запуск после первого + **−30 с** парткомпиляции = **−~210 с** | главный рычаг, бесплатно |
| Идемпотентный `install-ru.sh` (писать `.cfg` только если хэш реально сменился) | делает предыдущее автоматически | см. §6.4 |
| Перенести часть loc-патчей с `:FINAL` на `:FOR[kerbalru]`/`:AFTER[<mod>]`, убрать `@PART[*]` где не нужно | **−10..25 с** холодного перепатча | осторожно: `:FINAL` иногда нужен, чтобы перебить loc RO/RP-1 — проверять по одному |
| RSS-Textures 8K→4K | −5..10 с загрузки, **−1.2 ГБ RAM** | загрузку почти не ускоряет (8K грузятся OnDemand по подлёту), это мера памяти |
| Убрать битый `KerbalChangelog` (`ArgumentException: Invalid version 0.43.0.11b`) | ~0 с | чистит EXC, косметика/стабильность |

### 4.4. Про 8K-текстуры и «поможет ли 4K загрузке»

8K-поверхности (`EarthSurface.dds` и т.п., папка 1.6 ГБ) грузятся через **Kopernicus OnDemand** —
в логе `[OD] ... enabling self` стоит в 23:54:57, **после** 359 с меню. Значит в 360 с загрузки
8K-поверхности **почти не входят**. 4K → загрузку ускорит слабо (−5..10 с), но даст **−1.2 ГБ
RAM/VRAM** и меньше фризов при смене тела. **4K — мера памяти/стабильности, не времени.**

Проверь `Kopernicus/Config/Kopernicus_Config.cfg → UseOnDemandLoader = True` (в свежем Kopernicus
по умолчанию True; при False все 8K грузятся на старте).

### 4.5. Практический потолок

- **Холодный запуск после установки/обновления русификатора: ~330–360 с — неизбежно.** MM обязан
  перепатчить и пересобрать ConfigCache; 185 с для 78181 патча single-thread не ускорить ничем,
  кроме уменьшения числа патчей/модов.
- **Тёплый (кэш валиден, `.cfg` не менялись): ~150 с (−210 с).**
- **Тёплый + 4K + прореженные визуалки: ~110–130 с.**

## 5. Платформенное

### 5.1. Mac (M1 Pro, Rosetta) — вердикт по Whisky/CrossOver

**Стоит попробовать как эксперимент ради in-flight FPS, но это НЕ обязательный шаг инсталлятора и
НЕ панацея.**

За: Windows-сборка рендерит через **DX11 → D3DMetal (напрямую в Metal)** или DXVK→MoltenVK→Metal.
Metal-путь кардинально быстрее дохлого Apple OpenGL — отсюда отзывы про «огромный прирост FPS» в
GPU-сценах (вид на поверхность). RO/RP-1 — чистый managed C#/Unity без анти-чита → под Wine
запускаются штатно. Community: CrossOver (D3DMetal/DXVK) > Whisky по скорости.

Против:
- **6-мин загрузку это НЕ лечит** — MM-патчинг и физика остаются CPU/Rosetta; под Whisky идут
  через Rosetta **плюс** тонкий Wine-слой → в лучшем случае так же.
- **Тот же баг DXVK GTT-overspill (#5140)** ударит и на Mac при 8K RSS → **сначала ужать текстуры
  до 2K**, потом тестировать.
- CrossOver платный (~$74); **Whisky бесплатный, но разработка свёрнута** (2024–25), DXVK у него
  медленнее. У пользователя Whisky уже стоит — начинать с него.
- Parallels смысла не имеет: полная VM Windows на ARM + эмуляция x86 внутри = медленнее CrossOver.

Протокол A/B (если тестировать): (1) сначала ужать текстуры до 2K + срезать шейдеры — база для
честного сравнения и обход overspill; (2) в Whisky Windows-сборка с **D3DMetal**, та же полётная
сцена; (3) A/B против нативного Rosetta по FPS. Ожидание: GPU-сцены выиграют на Whisky; загрузка и
физика — паритет или чуть хуже. **Если визуал-моды уже отключены — разрыв сокращается, смысла в
Whisky меньше.**

→ **В автоустановку Whisky-путь НЕ автоматизировать** — оставить опциональной заметкой в docs.

**Гигиена Mac (16 ГБ в обрез):** закрыть браузеры/Electron/Docker перед запуском; ноутбук на
зарядке + «Высокая производительность» (System Settings → Battery); внешнее охлаждение (троттлинг
M1 Pro в длинной CPU-сессии реален). Launch-опции `-force-glcore/-force-opengl/-force-metal` на
Mac **не дают Metal и не помогают** — не карго-культить.

### 5.2. Steam Deck — Proton / FSR / питание

Ранжировано по эффекту:

1. **Текстуры 8K→2K — критично** (обходит баг GTT-overspill #5140, см. §2.3). Первый шаг.
2. **Ставить на внутренний NVMe, НЕ на microSD** — загрузка ассетов I/O-bound; microSD ~в 3–4
   раза медленнее.
3. **Отключить тяжёлые визуалки** (Scatterer океан off, EVE `layerVolume` вырезать,
   `SHADOWS_QUALITY`→0/1) — для 15 Вт APU это разница между слайд-шоу и играбельностью.
4. **Профиль питания** (Quick Access → Performance):
   - **Кап FPS = 30** (LCD 60 Гц) или 30/40 (OLED) — в сложных сценах KSP всё равно не выдаст
     больше; капом убираем judder и жор батареи.
   - **TDP = 15 Вт** (максимум) — игре нужен CPU-запас.
   - **Manual GPU Clock низкий (~400–600 МГц)** — известный трюк для CPU-bound игр: не даёт GPU
     отъедать общий power/термо-бюджет у CPU (после отключения визуалок GPU не нужен высокий).
   - **FSR через gamescope** — рендер в ~540–600p → апскейл в 800p; помогает, если оставили
     что-то из визуала. Half-rate shading в gamescope — ещё дешевле.
5. **Native Linux vs Proton:**
   - **Оставил крупные текстуры → native Linux безопаснее** (OpenGL-путь не ловит overspill #5140,
     хотя сам медленнее/однопоточнее).
   - **Ужал до 2K (рекомендация №1) → Proton (DXVK, DX11→Vulkan) эффективнее на RDNA2**, overspill
     не срабатывает. **Предпочтительный путь.** Для native — снять «Force compatibility Tool»; для
     Proton — Proton GE актуальнее стокового.
6. **Память:** SteamOS по умолчанию с zram; под RO/RSS стоит увеличить swap, чтобы пики не убивали
   OOM-килом. Вторично после ужатия текстур.

## 6. Дизайн интеграции в автоустановку kerbal.ru

Цель: **опциональный, по умолчанию выключенный** «профиль производительности», применяемый одной
командой. Выключен по умолчанию — потому что правки настроек субъективны и связаны с железом
(соответствует осторожности `FOR-AGENTS.md`: «не оптимизируй `settings.cfg` вслепую»).

### 6.1. Три канала доставки (по природе изменений)

| Канал | Что | Кто применяет | Легальность |
|---|---|---|---|
| **A. CKAN-свапы** | RSSTextures 8K→4K/2K; установка HeapPadder; снятие SXT/VenStock | тот же headless-CKAN, что уже в `install.sh` | чужие моды НЕ храним — вызываем CKAN |
| **B. Наш config-мод** | Scatterer/EVE MM-патчи, KSPCF-extras, padheap.cfg | копируем в GameData из репо | наши файлы — распространять можно |
| **C. Фрагмент settings.cfg** | стоковые графика/физика (§2.1–2.2) | merge в `settings.cfg` до запуска | стоковый файл игры, правим key-by-key |

### 6.2. Раскладка файлов в репозитории

```
perf/
  README.md                          # человекочитаемо: что делает профиль, как откатить
  settings/                          # канал C — фрагменты для merge в settings.cfg
    common.env                        #   key=value, общие для обоих профилей
    mac.env                           #   Mac-специфичные (SHADOWS_QUALITY=2, PHYSICS…=0.03…)
    deck.env                          #   Deck-специфичные (SHADOWS_QUALITY=1, FRAMERATE_LIMIT=30…)
  gamedata/                          # канал B — наши config-моды, копируются в GameData/
    kerbalru-perf-mac/
      kerbalru-perf.cfg               #   @KSPCommunityFixes:FINAL{...}, @Scatterer{...}, EVE-патчи (Mac-значения)
    kerbalru-perf-deck/
      kerbalru-perf.cfg               #   те же ноды, Deck-значения (океан off, layerVolume вырезан)
  heappadder/
    padheap.cfg                       #   наш урезанный пэддинг ~512–768 МБ (кладётся после CKAN-install мода)
  ckan-swaps.txt                     # канал A — список команд remove/install по профилям (данные, не код)
```

`kerbalru-perf-<profile>/` — обычный GameData-мод с одним `.cfg`. Install.sh копирует **ровно
один** вариант в `GameData/kerbalru-perf/` по выбранному профилю и **регистрирует его в manifest**
`.kerbalru-files` (чтобы `install-ru.sh` при обновлении/откате им управлял, как остальными
файлами).

### 6.3. Механика правки `settings.cfg` (канал C) — почему до запуска

KSP читает `settings.cfg` на старте, держит настройки в памяти и **перезаписывает файл на чистом
выходе**. Отсюда:
- **Патчим до первого запуска** → игра читает наши значения, держит их, на выходе пишет обратно
  те же → **сохраняются**. ✓
- **Патчим при закрытой игре** — если игра запущена, её выход затрёт наш патч.
- Если пользователь позже поменяет графику **в игре** — его выбор победит (записан на выходе). Это
  нормально, его машина.

Реализация — тем же приёмом, что уже применяется для `LANGUAGE = ru` в `install-ru.sh` (sed по
ключу + бэкап `settings.cfg.bak-kerbalru`), но по списку ключей из `perf/settings/*.env`:

```
для каждой строки KEY=VALUE из common.env + <profile>.env:
  если в settings.cfg есть "^KEY = " → заменить значение
  иначе → дописать "KEY = VALUE"
```

Бэкап: тот же `settings.cfg.bak-kerbalru` (уже создаётся установщиком) — гарантирует откат.

⚠️ **Грабля: `settings.cfg` может ещё не существовать** (свежий CKAN-инстанс до первого запуска
игры). Тогда канал C пропускается с сообщением: «настройки применятся при повторном
`--perf` после первого запуска игры». Каналы A и B при этом отрабатывают штатно. НЕ генерировать
`settings.cfg` с нуля (десятки недокументированных ключей — легко получить невалидный).

### 6.4. Флаг и поток управления

Расширяем существующий `install.sh` (он уже оркеструет CKAN + `install-ru.sh`):

```
install.sh [путь] [--full|--ru-only] [--perf[=auto|mac|deck|off]]
```

- **без `--perf`** — поведение как сейчас, ничего перф-специфичного (opt-in).
- **`--perf` / `--perf=auto`** — определить платформу (§6.5), применить соответствующий профиль.
- **`--perf=mac|deck`** — форсировать профиль (полезно для generic-Linux-десктопа: возьмёт `deck`
  как консервативный GPU-профиль без Deck-специфики питания).
- **`--perf=off`** — откатить: восстановить `settings.cfg.bak-kerbalru`, удалить
  `GameData/kerbalru-perf/`, (опц.) вернуть 8K-текстуры через CKAN.

Порядок применения профиля (важно для кэша MM — см. §4):
1. **CKAN-свапы (канал A)** — до русификатора: текстуры, HeapPadder, снятие SXT/VenStock.
2. **padheap.cfg** — поверх свежепоставленного HeapPadder.
3. **config-мод (канал B)** в `GameData/kerbalru-perf/` — **вместе** с копированием русификатора,
   одним заходом, чтобы MM пересобрал ConfigCache **ровно один раз** (а не отдельной инвалидацией).
4. **settings.cfg (канал C)** — последним, при закрытой игре.

Интерактив (когда терминал есть и флаг не задан) — по образцу существующего вопроса «1/2» в
`install.sh`: после установки предложить «применить профиль производительности для этой платформы?
[Y/n]». Для агента/CI предпочтителен явный `--perf=...`.

`install-ru.sh` тоже принимает `--perf=<profile>` (pass-through из `install.sh`), чтобы ru-only
путь мог применить профиль поверх уже стоящей сборки — но канал A (CKAN) в ru-only доступен только
если найдены `mono`+`ckan` (иначе печатает команды `ckan-swaps.txt` и выполняет только B и C).

### 6.5. Детект платформы

Опираемся на то, что уже есть в скриптах (`uname -s`, кандидатные пути с `/run/media/deck`):

```
uname_s = uname -s
если uname_s == Darwin:
    profile = mac                     # Apple Silicon доп. проверка: uname -m == arm64
если uname_s == Linux:
    если /etc/os-release содержит 'ID=steamos' или 'VARIANT_ID=steamdeck'
       ИЛИ путь KSP начинается с /run/media/deck или ~/.local/share/Steam на SteamOS
       ИЛИ есть steamos-session-select:
        profile = deck
    иначе:
        profile = deck (консервативный) с предупреждением, что это generic-Linux,
        либо спросить/ждать явный --perf=mac|deck
```

Deck-специфичные вещи (кап FPS, TDP, GPU-clock, FSR) — **не в файлах игры**, а в Game Mode Deck →
их автоустановка **не** трогает; кладём в `perf/README.md` как чеклист для пользователя (+пункт
роадмапа «один клик из Gaming Mode»).

### 6.6. Идемпотентность и MM-кэш (обязательное требование)

Чтобы профиль не превращал каждый запуск в 6 минут (§4.2):
- **`install-ru.sh` — идемпотентный:** сравнивать хэш перед записью каждого `.cfg`, не
  перезаписывать идентичное. Тогда MM-кэш переживает повторные прогоны.
- **Разделить каналы обновления:** `.txt`-словари катить свободно (кэш не трогают); `.cfg`-патчи
  (в т.ч. наш `kerbalru-perf.cfg`) — только при смене версии (одна инвалидация на релиз, а не на
  запуск).
- Профиль ставить **одним заходом** с русификатором (шаг 3 в §6.4) → MM пересобирает кэш один раз.
- В конце установщик печатает: «первый запуск после установки ~5–6 мин (сборка кэша MM),
  последующие ~2.5 мин» — снимает жалобу «хромает».

## Приложение: сводка CKAN-идентификаторов

| Действие | Команда |
|---|---|
| Текстуры 4K (Mac) | `ckan remove --headless RSSTextures8192 && ckan install --headless RSSTextures4096` |
| Текстуры 2K (Deck) | `ckan remove --headless RSSTextures8192 && ckan install --headless RSSTextures2048` |
| GC-фризы (авто) | `ckan install --headless HeapPadder` |
| GC-фризы (с графиком) | `ckan install --headless MemGraph` |
| Снять лишние парт-паки | `ckan remove --headless SXT VenStockRevamp` |

> Идентификаторы `RSSTextures2048/4096/8192`, `HeapPadder`, `MemGraph`, `SXT`, `VenStockRevamp`
> сверены по отчётам; **перед прошивкой в установщик перепроверить по живому CKAN-реестру**
> (`ckan search <id>`), особенно вариант RSSVE-LR и точное имя ноды Scatterer для MM-патча.
