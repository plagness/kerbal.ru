# Как участвовать в kerbal.ru

Спасибо, что хочешь сделать реалистичную сборку KSP понятнее русскоязычным игрокам. Здесь полезны переводчики, редакторы, тестировщики, разработчики и люди, которые просто умеют хорошо описывать ошибки.

Участие в проекте означает согласие с [кодексом поведения](CODE_OF_CONDUCT.md).

## С чего начать

1. Для вопроса или сырой идеи открой [Discussion](https://github.com/plagness/kerbal.ru/discussions).
2. Для воспроизводимого бага или конкретной задачи используй подходящий [issue-шаблон](https://github.com/plagness/kerbal.ru/issues/new/choose).
3. Для небольшого очевидного исправления можно сразу открыть pull request.
4. Для крупного перевода или изменения установщика сначала согласуй подход: так два человека не переведут один и тот же мод параллельно.

Хорошие первые задачи помечаются `good first issue`; задачи, где особенно нужен участник, — `help wanted`.

## Виды вкладов

### Перевод

- Выбери мод из [списка кандидатов](docs/COVERAGE.md#3b-реальные-кандидаты-для-перевода-есть-деталиui--17) или существующую issue.
- Используй терминологию реальной космонавтики.
- Не меняй оригинальные файлы мода: добавляй `ru.cfg` или отдельный ModuleManager-патч.
- Сохраняй плейсхолдеры, служебные теги и имена брендов.
- Следуй полным правилам из [MAINTAINING.md](docs/MAINTAINING.md#3-правила-перевода-обязательные).

### Проверка перевода

Структурно правильный `ru.cfg` ещё не доказывает, что игра показывает русский текст. Самая ценная проверка:

1. установить PR в тестовую KSP;
2. один раз запустить игру, чтобы ModuleManager пересобрал кэш;
3. выполнить `tools/audit_ru_coverage.py` с путём к игре;
4. приложить к PR результат и, если уместно, скриншот интерфейса.

### Код, сайт и установщики

- Сохраняй проект статическим и простым в публикации на GitHub Pages.
- Не добавляй проприетарные ассеты KSP.
- Для сайта проверяй desktop/mobile, отсутствие ошибок в консоли и режим `prefers-reduced-motion`.
- Для установщика обязательно проверяй повторный запуск: обновление не должно удалять чужие файлы из `GameData`.

### Документация

Исправления инструкций, FAQ, терминологии и опечаток можно отправлять напрямую. Команды должны быть копируемыми, пути — учитывать пробелы, а утверждения о покрытии — ссылаться на аудит.

## Локальная работа

```bash
git clone https://github.com/plagness/kerbal.ru.git
cd kerbal.ru
python3 -m http.server 4173
```

Открой <http://127.0.0.1:4173>. Для правок сайта достаточно современного браузера; сборщик не нужен.

## Проверки перед PR

```bash
# синтаксис Unix-установщика (менеджера сборок)
bash -n install.sh

# Python-инструмент
python3 -c "import ast,pathlib; ast.parse(pathlib.Path('tools/audit_ru_coverage.py').read_text())"

# пробелы и конфликтные маркеры
git diff --check
```

Для перевода дополнительно прогони `python3 tools/validate_localization.py` — он строит настоящее дерево `{}`-блоков (не просто считает скобки), ловит ключи, случайно оказавшиеся не в том блоке, дубликаты, рассинхрон en-us/ru и «зависшие» ссылки на несуществующие ключи. Плюс живой `ModuleManager.ConfigCache` после реального запуска игры — структурная проверка не гарантирует, что текст резолвится (см. [MAINTAINING.md, раздел 6](docs/MAINTAINING.md#6-грабли-реально-пойманные-важно)). Автоматические проверки также запустятся в GitHub Actions на каждый PR.

Для правки словарей русификации интерфейса модов (`translations/<mod>/KerbalRuUiTranslations/<Mod>.txt`) прогони `python3 tools/ui-translator/validate_dict.py` — проверяет формат, дубликаты ключей и конфликты перевода одного и того же английского текста между модами. Подробности техники и известные ограничения — [docs/UI-TRANSLATION.md](docs/UI-TRANSLATION.md).

## Pull request

- Один PR — одна понятная цель.
- Заголовок описывает результат, а не процесс: `Перевести UI RealAntennas`, а не `Правки файлов`.
- В описании укажи, что изменено, зачем, как проверено и есть ли ограничения.
- Не включай личные настройки, логи целиком, кэши игры или файлы самой KSP.
- Изменение перевода должно назвать мод, его версию и лицензию источника.
- Не переписывай историю ветки после начала ревью без необходимости.

Maintainer может попросить сократить перевод, исправить термин или подтвердить поведение живым запуском. Это часть проверки, а не оценка участника.

## После принятия

Изменения попадают в `main`, проходят проверки и становятся частью следующего стабильного релиза. Сайт публикуется GitHub Pages автоматически. Версия перевода меняется только отдельным релизным PR; правила выпуска — в [GOVERNANCE.md](GOVERNANCE.md#релизы).
